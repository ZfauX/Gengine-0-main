# ADR-001: Domain-Driven Package Structure

**Статус**: Принято

**Контекст**: Проект быстро рос, потребовалась единообразная структура для всех функциональных модулей. Без стандартизации каждый разработчик мог организовывать код по-своему, что усложняло навигацию и поддержку.

**Решение**: Каждый домен (user, game, team, level, tournament, monitor, calendar, social, notification, admin, export) следует единому шаблону:
- `model.go` — GORM-модели, типы, константы
- `service.go` — бизнес-логика (чистый Go, без HTTP/HTML)
- `handler.go` — HTTP-обработчики (gin.Context, формы, шаблоны)
- `routes.go` — регистрация маршрутов с DI и middleware
- `templates/` — HTML-шаблоны для домена

**Последствия**: 
- Предсказуемая навигация по проекту
- Новый домен добавляется копированием шаблона
- Чёткое разделение ответственности: сервисы не знают о HTTP
- Дублирование типов между доменами (например, `gameLevel` и `level.Level`)

# ADR-002: Dependency Injection via google/wire

**Статус**: Принято

**Контекст**: Ручное конструирование зависимостей в `init.go` (27 сервисов, 17 репозиториев) стало хрупким — добавление нового параметра в конструктор требовало ручного обновления всего графа зависимостей. Это приводило к ошибкам компиляции только в рантайме.

**Решение**: Внедрён `google/wire` для кодогенерации DI. Определены две функции-инжектора:
- `initializeRepositories(db)` — собирает 17 репозиториев
- `initializeServices(db, repos, cfg, hub, storage, cache)` — собирает 27 сервисов

`wire_providers.go` содержит функции-обёртки для сервисов с method-chaining (`.WithSSEManager()`, `.WithGameService()`).

**Последствия**:
- Компиляторная проверка графа зависимостей
- `//go:generate go run github.com/google/wire/cmd/wire` в `init.go`
- При изменении конструктора достаточно запустить `go generate ./internal/app/`
- Дополнительная сложность: wire не поддерживает method-chaining напрямую

# ADR-003: PostgreSQL + GORM with golang-migrate

**Статус**: Принято

**Контекст**: Требовалась надёжная, транзакционная БД с поддержкой сложных запросов (оконные функции, полнотекстовый поиск, блокировки строк). Выбор между SQLite (простота) и PostgreSQL (функциональность).

**Решение**: PostgreSQL как основная БД, GORM как ORM, `golang-migrate` для версионированных миграций (11 миграций: 000001–000011).

Ключевые паттерны:
- `testutil.SetupPostgresDB` — изолированные схемы для параллельных тестов
- `clause.Locking{Strength: "UPDATE"}` — блокировка строк в конкурентных сценариях
- `plainto_tsquery('russian', ...)` — полнотекстовый поиск
- Пул соединений: `MaxOpenConns: 25-50`, `MaxIdleConns: 10-25`

**Последствия**:
- Тесты требуют запущенного PostgreSQL
- GORM скрывает нюансы SQL (N+1, eager loading)
- `AutoMigrate` + `MigrateFromFiles` — двойственность, возможна рассинхронизация

# ADR-004: SSE + WebSocket для реал-тайм уведомлений

**Статус**: Принято

**Контекст**: Требовалось два типа реал-тайм коммуникации: 1) однонаправленные уведомления для игроков (старт игры, прохождение уровня), 2) двунаправленный чат для мониторинга и поддержки.

**Решение**:
- **SSE** (Server-Sent Events) — для уведомлений: `SSEManager` управляет пулом сессий по `gameID`, `Broadcast()` рассылает события всем подписчикам игры
- **WebSocket** — для чата: `RoomHub` c регистрацией/отпиской, комнатами по `gameID`, rate limiting по IP

SSE выбран вместо WebSocket для уведомлений, потому что:
- Проще (однонаправленный протокол)
- Авто-переподключение через браузерный `EventSource`
- Не требует отдельного WS-сервера

**Последствия**:
- Два параллельных механизма реал-тайм (SSE + WS)
- SSE не поддерживает кастомные заголовки → нет CSRF-токена, нужна Origin-проверка
- WebSocket требует handshake-роутинга и heartbeat

# ADR-005: JWT Cookie Authentication

**Статус**: Принято

**Контекст**: Требовалась аутентификация для SSR-приложения без SPA. Сессионные cookie (память на сервере) VS JWT (stateless).

**Решение**: 
- JWT access-токен (15 мин) + refresh-токен (30 дней, в БД)
- JWT хранится в httpOnly cookie (недоступен JS)
- Refresh-токен в БД позволяет отзывать сессии (logout-all)
- Middleware: `AuthRequired` (редирект на `/auth/login`) и `OptionalAuth` (продолжает без userID)
- 2FA через TOTP + backup-коды, middleware `TwoFactorRequired`

**Последствия**:
- httpOnly cookie защищает от XSS-кражи токена
- Refresh-токены в БД — дополнительная нагрузка на запись
- Нет CSRF на API-эндпоинтах (только на HTML-формах через `gin-csrf`)
- 2FA middleware проверяется на каждом запросе (overhead)

# ADR-006: Two-Level Cache (Valkey + In-memory)

**Статус**: Принято

**Контекст**: Некоторые данные (рейтинги, снапшоты монитора, метрики) требовали кеширования для снижения нагрузки на БД. Выбор между in-memory cache (простота, нет сети) и Valkey/Redis (shared между инстансами).

**Решение**: Двухуровневая система:
1. **In-memory cache** (go-cache) — первичный уровень, TTL + cleanup interval
2. **Valkey/Redis** — вторичный уровень, JSON-сериализация, пул соединений
3. `NoopCache` — fallback при недоступности обоих

Кешируется: главная страница (`Cache-Control: public, max-age=60`), рейтинги, снапшоты монитора, метрики, rate limiter state.

**Последствия**:
- Graceful degradation при недоступности Valkey
- Non-`WithCtx` методы всё ещё используют `context.Background()` — потеря cancellation
- Rate лимитеры используют Valkey для shared state между инстансами
- In-memory cache не синхронизируется между инстансами (подходит для single-instance)

# ADR-007: Feature Flags через env-переменные

**Статус**: Принято

**Контекст**: Необходимость включать/выключать экспериментальные фичи без перекомпиляции и деплоя. Feature flags должны быть простыми и не требовать внешних зависимостей.

**Решение**: Пакет `internal/pkg/feature` с `IsEnabled(flag FeatureFlag) bool`, читающий переменные окружения `FEATURE_<NAME>=true`. Кеширование через `sync.Map`.

Флаги:
- `FEATURE_NEW_SSE` — новая реализация SSE
- `FEATURE_NEW_UI` — обновлённый интерфейс
- `FEATURE_EXPERIMENTAL_API` — экспериментальные API
- `FEATURE_STRICT_CONFIG` — строгая валидация (crash на неверные env)
- `FEATURE_TELEMETRY` — сбор телеметрии

**Последствия**:
- Zero external dependencies
- Переменные окружения видны во всех процессах (в т.ч. фоновых)
- Нет динамического переключения без рестарта (только env)
- Кеш сбрасывается только через `Reset()` (для тестов)
