## Как сообщить об ошибке или предложить идею

### 🐛 Сообщение об ошибке

Перед созданием Issue проверьте, нет ли уже похожей проблемы в [списке](https://github.com/ZfauX/Gengine-0/issues). Если нет — создайте новый Issue и по возможности укажите:

- **Версия** (commit, тег или ветка).
- **Описание** — что произошло и что ожидалось.
- **Шаги воспроизведения** — пошаговая инструкция, как повторить ошибку.
- **Окружение** — ОС, версия Go, БД (PostgreSQL/SQLite), переменные окружения (без секретов).
- **Логи** — если есть, приложите фрагменты логов с уровнем `debug` или `trace`.
- **Скриншоты** — если ошибка связана с интерфейсом.

### 💡 Предложение идеи

Новые фичи обсуждаются через Issue с меткой `enhancement`. Опишите:

- Какую проблему решает ваша идея.
- Как вы видите реализацию (архитектуру, новые эндпоинты, изменения в БД).
- Если есть примеры из других платформ — приложите ссылки.

## Подготовка окружения

Для локальной разработки вам понадобятся:

- Go 1.22+ (рекомендуется 1.23 или новее).
- PostgreSQL 14+ (или SQLite для юнит-тестов).
- (Опционально) Redis для кэширования сессий (не обязателен для запуска).
- (Опционально) `golangci-lint` для проверки стиля кода.

Клонируйте репозиторий:

```bash
git clone https://github.com/ZfauX/Gengine-0.git
cd Gengine-0

Скопируйте .env.example в .env и настройте обязательные переменные (см. README).

Рабочий процесс
Мы используем подход GitHub Flow:

Форкните репозиторий (если у вас нет прав на прямую запись).

Создайте ветку с говорящим названием:

feature/краткое-описание — для новой функциональности.

fix/краткое-описание — для исправления ошибки.

docs/краткое-описание — для обновления документации.

chore/краткое-описание — для служебных задач (обновление зависимостей, CI и т.п.).

Коммиты пишите на русском или английском (единообразно) в формате:

text
<тип>(<область>): <краткое описание>
Где <тип> — feat, fix, docs, chore, refactor, test, style и т.д.
Пример: feat(game): add blackbox voting for levels.
В теле коммита можно подробно описать изменения.

Регулярно синхронизируйтесь с основной веткой (main), чтобы избежать конфликтов:

bash
git checkout main
git pull upstream main
git checkout feature/your-branch
git rebase main
Запустите все тесты и проверьте линтер перед отправкой PR.

Создайте Pull Request в ветку main исходного репозитория.

Стиль кода и соглашения
Общие правила
Используйте gofmt и go vet — они обязательны.

Для статического анализа мы рекомендуем golangci-lint (конфигурация лежит в корне, если есть).
Запуск: golangci-lint run ./....

Все экспортируемые типы, функции, константы и переменные должны иметь godoc-комментарии на английском языке (или русском, если проект внутренний, но для open-source предпочтительнее английский).

Именование:

Пакеты — в нижнем регистре, короткие, осмысленные (например, game, user, monitor).

Экспортируемые идентификаторы — начинаются с заглавной буквы (CamelCase).

Неэкспортируемые — camelCase или snake_case (в Go принято camelCase для переменных, PascalCase для типов).

Константы — camelCase или UPPER_SNAKE_CASE (в Go часто camelCase).

Обработка ошибок:

Всегда обрабатывайте ошибки. Не используйте _ для игнорирования, если только это не действительно оправдано (например, в defer).

Используйте errors.New или fmt.Errorf с обёрткой (%w), чтобы сохранять цепочку.

В логировании используйте log.Error().Err(err).Msg(...).

Контекст — передавайте context.Context первым аргументом во все функции, которые могут делать сетевые вызовы или работать с БД.

Избегайте panic — используйте возврат ошибок. panic допустим только при инициализации (например, в main при фатальной ошибке).

Чистота кода — функция должна делать одно действие и быть не слишком длинной (<= 50 строк, в идеале). Если функция разрослась, разбейте её на вспомогательные.

Специфика проекта
Работа с БД — используйте GORM, но предпочтительнее писать запросы через db.Where(...).Find(...) и т.д., избегая сырых SQL (кроме сложных выборок).

Хендлеры Gin — разделяйте валидацию, логику и рендеринг. Валидацию делайте через ShouldBind, а затем проверяйте дополнительные условия.

Логирование — используйте zerolog с уровнями Debug, Info, Warn, Error. В production логи пишутся в JSON; в разработке можно использовать ConsoleWriter.

Тесты — для работы с БД используйте SQLite in-memory или транзакции с откатом.

Тестирование
Юнит-тесты
Пишите тесты для всех новых функций и исправлений.

Используйте пакет testing и библиотеку testify (assert/require).

Тесты должны быть независимыми и выполняться в изолированном окружении.

Для mock'ов используйте интерфейсы (например, репозитории) и генерируйте моки через mockgen или пишите вручную.

Запуск всех тестов:

bash
go test ./...
Интеграционные тесты
Для тестов, требующих реальной БД (PostgreSQL), используйте теги сборки или отдельные файлы с суффиксом _integration_test.go.
Пример запуска:

bash
go test -tags=integration ./internal/domain/...
Покрытие кода
Старайтесь поддерживать покрытие на уровне не менее 80% для критичных пакетов (game, user, level, team).
Проверка покрытия:

bash
go test ./internal/domain/... -coverprofile=coverage.out
go tool cover -html=coverage.out
Документация
Все экспортируемые типы, функции и константы должны быть задокументированы в формате godoc.

Если вы добавляете новый эндпоинт, обновите Swagger-аннотации в хендлерах (файлы handler.go).

При изменении конфигурации обновите .env.example и раздел "Configuration" в README.

Для новых фич опишите их в README (в разделе "Возможности" или отдельно).

Процесс Pull Request
Создайте PR с понятным заголовком и описанием.

В описании укажите:

Ссылку на Issue (если есть), например: Closes #123.

Список изменений (кратко).

Если есть миграции БД, опишите их.

Результаты тестов (если CI не запущен автоматически).

Проверьте CI — все проверки (тесты, линтер, сборка) должны быть зелёными.

Запросите ревью у maintainers (или оставьте комментарий, чтобы они обратили внимание).

После получения замечаний — исправьте их и обновите PR (добавьте новые коммиты или сделайте force-push после rebase).

Когда PR одобрен и все проверки пройдены, он будет смержен в main.

Лицензия и DCO
Этот проект распространяется под лицензией MIT (см. файл LICENSE).
При отправке Pull Request вы соглашаетесь с тем, что ваш код будет опубликован под этой лицензией.

Если проект использует Developer Certificate of Origin (DCO), убедитесь, что каждый коммит содержит Signed-off-by (можно добавить автоматически через git commit -s).
Проверьте, требуется ли это в CI.

Контакты
Если у вас возникли вопросы, вы можете:

Открыть Issue с меткой question.

Написать в обсуждения (если включены).

Связаться с мейнтейнерами через email (указан в профиле GitHub).

Благодарим за ваш вклад и желаем приятного кодинга! 🚀
